# ROS 2 - Robot Operating System

## Learning Objectives

- Learn how to create nodes, topics, services, and actions

## Introduction to ROS 2

ROS 2 (Robot Operating System 2) is a flexible framework for writing robot software. It's a collection of tools, libraries, and conventions that aim to simplify the task of creating complex and robust robot behavior across a wide variety of robot platforms. Unlike traditional operating systems, ROS 2 is middleware that provides services designed specifically for robotic applications.

ROS 2 is the successor to the original Robot Operating System (ROS 1), addressing many of its predecessor's limitations, particularly in areas like security, real-time performance, and multi-robot systems.

## Key Concepts in ROS 2

### Nodes
A node is an executable that uses ROS 2 to communicate with other nodes. Nodes are typically written in C++ or Python, but bindings exist for other languages. Each node can subscribe to or publish data to different topics, provide or use services, and send or receive actions.

### Packages
Packages are the software containers in ROS 2. Each package can contain libraries, executables, configuration files, and other resources needed to provide a specific functionality. A package may contain one or more nodes.

### Topics and Messages
Topics are communication channels through which nodes send and receive messages. Messages are data structures that are passed between nodes. Each message has a specific type with a defined field structure.

### Services
Services provide a request/response communication pattern. When a node calls a service, it sends a request and waits for a response. This is useful for operations that have a clear input/output relationship and are expected to complete relatively quickly.

### Actions
Actions are similar to services but are designed for long-running tasks. They provide feedback during execution and can be preempted (canceled) before completion. Actions are ideal for goals that take a significant amount of time, like navigation or manipulation tasks.

## ROS 2 Architecture

### Client Libraries
ROS 2 provides client libraries that enable nodes written in different languages to communicate seamlessly. The most commonly used libraries are rclcpp for C++ and rclpy for Python.

### Middleware and DDS
ROS 2 uses a middleware layer based on Data Distribution Service (DDS) to enable communication between nodes. DDS handles the underlying network communication, allowing nodes to communicate across different machines and platforms.

### Quality of Service (QoS) Settings
QoS policies allow fine-tuning of the communication behavior between nodes. They include settings such as reliability (reliable vs. best-effort), durability (transient vs. volatile), and deadline constraints.

## Development Environment

### Workspaces
ROS 2 uses workspaces to organize packages. A workspace is a directory that contains packages, build directories, and install directories. The standard structure follows the colcon build system.

### Build System (colcon)
Colcon is the ROS 2 build system that builds packages in the correct order based on their dependencies. It supports building packages in parallel for faster compilation.

## Essential ROS 2 Tools

### ros2 run and ros2 launch
- `ros2 run` executes a single executable from a package
- `ros2 launch` starts multiple nodes simultaneously using launch files

### ros2 topic
This command allows inspection and interaction with topics:
- `ros2 topic list`: List active topics
- `ros2 topic echo`: Print messages published to a topic
- `ros2 topic pub`: Publish messages to a topic

### ros2 service
This command interacts with services:
- `ros2 service list`: List active services
- `ros2 service call`: Call a service with specified parameters

### rviz2
RViz2 is the 3D visualization tool for ROS 2. It can display sensor data, robot models, paths, and other information in a 3D environment.

### rqt
RQt is a GUI framework for ROS 2 that provides various plugins for visualizing data, monitoring performance, and debugging nodes.

## ROS 2 in Physical AI Systems

### Multi-Robot Systems
ROS 2 is well-suited for multi-robot systems:
- Each robot can run its own ROS 2 domain
- Communication between robots can happen through various means (DDS, bridge nodes, etc.)
- Coordination can be handled through distributed systems

### Real-time Performance
ROS 2 provides better real-time support than ROS 1:
- Improved determinism in communication
- Support for real-time operating systems
- Configurable QoS settings for time-critical applications

### Security
ROS 2 includes security features important for Physical AI:
- Authentication of nodes
- Encryption of communication
- Authorization of network access

### Lifecycle Management
ROS 2 provides tools for managing node lifecycles, allowing for more robust systems:
- Nodes can transition between different states (unconfigured, inactive, active)
- Supervisors can manage groups of nodes
- Error recovery and restart capabilities

## Best Practices

### Package Structure
Follow the standard ROS 2 package structure:
- Use meaningful names for packages
- Separate concerns into different packages
- Use dependency management effectively

### Communication Design
- Use topics for data streams
- Use services for request/response interactions
- Use actions for long-running tasks with goals
- Design custom message types for specific applications

### Testing and Debugging
- Write unit tests for nodes
- Use launch files for integration testing
- Implement logging at appropriate levels
- Design for observability using ROS 2 tools
